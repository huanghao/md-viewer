# 文档治理规范

## 核心原则：内容活性

**文档的存在理由是"当前有人会读它"，而不是"当时写了它"。**

腐烂的文档比没有文档更危险——它误导 agent 和未来的自己。每隔一段时间问自己：这个文档描述的还是现实吗？答不上来就删。

## 1. 文档分层

| 层级 | 目录/文件 | 说明 |
|------|-----------|------|
| 项目规范 | `README.md`、`AGENTS.md` | 长期维护，必须与现实同步 |
| 设计文档 | `docs/design/` | 功能设计，实现后仍可作为设计档案 |
| 专题文档 | `docs/` 根目录 | 独立主题，需明确读者和时效 |
| 调研记录 | `docs/research/`、`docs/testing/` | 阶段性，不强制维护 |
| AI 设计规范 | `docs/superpowers/specs/` | superpowers 生成的设计文档 |

**没有归档目录。** 不再用的文档直接删，不迁移到 archive。

## 2. 删除优先于保留

以下类型的文档**应当删除**，不是归档：

- **实现完成的 plan 类文档**：代码和 git log 是唯一的完成证明，plan 文档在功能上线后立即失去价值
- **静态快照型文档**：如用户风格、协作模式说明——这类信息应存在于动态系统（memory、代码、对话）中，写成文档就会腐烂
- **被工具/流程替代的文档**：如 man 手册（--help 够用）、多 agent 协作规则（直接开 terminal）
- **重复内容**：同一事实只能有一个权威位置，其他地方引用而不复制

## 3. 单一事实源

- 同一主题只保留 1 份权威文档，其他地方只引用，不复制参数表、状态规则、流程描述
- CLI 用法的真相源是 `--help` 输出，不另立文档
- 用户偏好的真相源是 memory 系统，不写成静态文档

## 4. 文档命名

- 权威单份文档用稳定文件名，不加日期前缀
- 设计/调研/阶段性文档用 `YYYYmmdd-` 前缀
- 同一主题多个文件（设计文档 + 原型 + 素材）放进主题目录 `docs/design/YYYYmmdd-<topic>/`

## 5. 变更同步

CLI 参数或配置行为变化时，同步更新：
- `showHelp()` 输出（真相源）
- 对应的设计文档（如有）

## 6. 巡检清单

```
[ ] 有没有描述已不存在的功能/流程的文档？→ 删除
[ ] 有没有重复描述同一件事的文档？→ 合并或删除副本
[ ] 有没有实现完成但 plan 文档还留着的？→ 删除 plan
[ ] docs/README.md 的索引是否还准确？→ 更新
[ ] AGENTS.md 引用的文件是否都存在？→ 修正
```
