# 划词评论功能设计

## 需求背景

Agent 会生成很多设计文档，用户需要：
1. 对文档中的特定内容提问
2. 确认或否决某个方案
3. 添加补充说明
4. 多轮对话直到收敛

**目标：标准化这个协作流程，提升效率和质量。**

## 业界参考

### 1. **Google Docs** - 评论 + 建议模式
- 选中文字 → 右键 → 添加评论
- 评论显示在侧边栏
- 支持回复、解决、重新打开
- 支持 @ 提及他人

### 2. **Notion** - 讨论功能
- 选中文字 → 添加讨论
- 讨论以气泡形式显示
- 支持回复、解决
- 解决后折叠

### 3. **Figma** - 评论钉
- 点击位置添加评论
- 评论以数字标记显示
- 点击查看完整对话
- 支持解决、删除

### 4. **GitHub** - PR Review Comments
- 行级评论
- 支持建议修改
- 支持批注、回复
- 可解决对话

### 5. **Medium** - 高亮 + 注释
- 选中文字自动高亮
- 侧边栏显示注释
- 支持公开/私有注释

## 设计方案

### 方案 A：Markdown 注释语法（推荐）

**原理：**
在 Markdown 文件中使用特殊语法嵌入评论，既保持纯文本可读性，又能被工具解析。

**语法设计：**
```markdown
<!-- @comment:user:timestamp
Q: 这个方案的性能如何？
-->

<!-- @reply:agent:timestamp
A: 性能测试显示可以处理 10000+ 文件，CPU 占用 < 5%
-->

<!-- @resolved:timestamp
已确认性能满足需求
-->
```

**优点：**
- ✅ 纯文本，Git 友好
- ✅ 不破坏 Markdown 渲染
- ✅ 可以在任何编辑器中查看
- ✅ Agent 可以直接解析和回复

**缺点：**
- ⚠️ 需要手动写语法
- ⚠️ 视觉上不够直观

---

### 方案 B：YAML Front Matter + 评论区

**原理：**
在文档开头使用 YAML Front Matter 存储评论元数据。

**格式：**
```markdown
---
comments:
  - id: c1
    user: huanghao
    text: "方案 A 和方案 B 哪个更好？"
    target: "## 方案 A"
    timestamp: 2026-03-01T02:00:00Z
    replies:
      - user: agent
        text: "方案 A 更简单，推荐使用"
        timestamp: 2026-03-01T02:01:00Z
    status: resolved
---

# 设计文档

## 方案 A
...
```

**优点：**
- ✅ 结构化，易解析
- ✅ 不影响正文
- ✅ Git 友好

**缺点：**
- ⚠️ 评论和内容分离，不直观
- ⚠️ 修改正文后，target 可能失效

---

### 方案 C：侧边栏评论（UI 实现）

**原理：**
在 MD Viewer 中添加评论功能，评论存储在独立文件中。

**实现：**
```
docs/design/logo-redesign.md        # 文档
docs/design/.comments/logo-redesign.json  # 评论数据
```

**评论数据格式：**
```json
{
  "comments": [
    {
      "id": "c1",
      "user": "huanghao",
      "text": "方案 A 和方案 B 哪个更好？",
      "selection": {
        "start": 120,
        "end": 150,
        "text": "## 方案 A"
      },
      "timestamp": "2026-03-01T02:00:00Z",
      "replies": [
        {
          "user": "agent",
          "text": "方案 A 更简单，推荐使用",
          "timestamp": "2026-03-01T02:01:00Z"
        }
      ],
      "status": "resolved"
    }
  ]
}
```

**UI 设计：**
```
┌─────────────────────────────────────────────────────┐
│  文档 / 设计 / logo-redesign.md                     │
├─────────────────────────────────────────────────────┤
│                                    │                │
│  # Logo 重新设计                   │  💬 评论 (2)   │
│                                    │                │
│  ## 方案 A [1]                     │  [1] huanghao  │
│  圆形徽章 + 简化字母 M            │  方案 A 和 B   │
│                                    │  哪个更好？    │
│  ## 方案 B                         │                │
│  方形 + 渐变背景                   │  → agent      │
│                                    │  方案 A 更简单 │
│                                    │                │
│                                    │  ✓ 已解决      │
└─────────────────────────────────────────────────────┘
```

**交互：**
1. 选中文字 → 右键 → "添加评论"
2. 弹出输入框，输入评论
3. 评论显示在侧边栏
4. 点击评论，高亮对应文字
5. 可回复、解决、删除

**优点：**
- ✅ 视觉直观
- ✅ 交互友好
- ✅ 支持富文本
- ✅ 评论和文档分离，不污染正文

**缺点：**
- ⚠️ 需要前端实现
- ⚠️ 评论数据在独立文件中
- ⚠️ Agent 需要读取评论文件

### 方案 D：GitHub Issue 集成

**原理：**
使用 GitHub Issue 作为评论系统。

**流程：**
1. Agent 生成设计文档并提交
2. 自动创建 GitHub Issue，标题为文档名
3. 用户在 Issue 中评论
4. Agent 监听 Issue，回复评论
5. 达成一致后，关闭 Issue

**优点：**
- ✅ 无需开发，直接用 GitHub
- ✅ 支持 @ 提及、标签、里程碑
- ✅ 有完整的通知系统
- ✅ 可追溯历史

**缺点：**
- ⚠️ 评论和文档分离
- ⚠️ 需要切换到浏览器
- ⚠️ 依赖 GitHub

## 推荐方案：A + C 组合

### 实施方案

#### 阶段 1：Markdown 注释语法（P0）

**立即可用，无需开发：**

```markdown
# Logo 重新设计

## 方案 A：圆形徽章
<!-- @comment:huanghao:2026-03-01
方案 A 和方案 B 哪个更好？
-->

<!-- @reply:agent:2026-03-01
方案 A 更简单，推荐使用。理由：
1. 圆形更柔和
2. 小尺寸下更清晰
3. 符合现代设计趋势
-->

<!-- @comment:huanghao:2026-03-01
确认使用方案 A
@resolved
-->

## 方案 B：方形 + 渐变
...
```

**Agent 处理流程：**
1. 读取文档，解析 `@comment`
2. 生成回复，插入 `@reply`
3. 用户确认后，添加 `@resolved`

**优点：**
- ✅ 立即可用
- ✅ Git 友好
- ✅ 纯文本
- ✅ Agent 可直接处理

## 标准化协作流程

### 1. Agent 生成设计文档

```markdown
# 功能设计

## 方案 A
...

## 方案 B
...

## 推荐
我推荐方案 A，请确认。

<!-- @comment:agent:auto
@question: 请选择方案 A 或方案 B
@options: A, B, 需要更多信息
-->
```

### 2. 用户添加评论

```markdown
<!-- @reply:huanghao:2026-03-01
选择方案 A，但需要调整颜色
-->
```

### 3. Agent 回复

```markdown
<!-- @reply:agent:2026-03-01
好的，我会调整颜色为 #3b82f6（更柔和的蓝色）
-->
```

### 4. 用户确认

```markdown
<!-- @comment:huanghao:2026-03-01
确认，可以实施
@resolved
@action: implement
-->
```

### 5. Agent 实施

Agent 检测到 `@action: implement`，开始实施方案 A。

## 评论语法规范

### 基础语法
```markdown
<!-- @comment:user:timestamp
评论内容
-->
```

### 回复语法
```markdown
<!-- @reply:user:timestamp
回复内容
-->
```

### 解决语法
```markdown
<!-- @resolved:timestamp
解决说明
-->
```

### 标签语法
```markdown
<!-- @comment:user:timestamp
@question: 这是一个问题
@options: A, B, C
@priority: high
-->
```

### 提及语法
```markdown
<!-- @comment:user:timestamp
@agent 请回答这个问题
-->
```
