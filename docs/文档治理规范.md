# 文档治理规范

本文档定义 `md-viewer` 的文档整理与长期维护规则，目标是减少重复、控制过时信息、降低维护成本。

## 1. 文档分层

1. 长期规范文档（主路径）：
   - `README.md`
   - `AGENTS.md`
   - `man/*.1`、`man/*.5`
   - `docs/配置说明.md`
   - 持续维护的功能说明/设计规范
2. 阶段性文档（工作路径）：
   - 调研记录
   - 阶段性测试记录
   - 设计评审记录
3. 进行中目录：
   - `docs/design/`
   - `docs/testing/`
   - `docs/research/`
   - `docs/brainstorming/`
4. 归档目录：
   - `docs/archive/YYYY-MM/`

## 2. 单一事实源规则

1. 同一主题只保留 1 份“权威文档”。
2. 其余文档只引用权威文档，不复制参数表、状态规则、流程描述。
3. 每份文档开头建议声明“文档定位”（如：用户、设计、测试、证据、归档）。
4. 命名规则：
   - 权威单份文档保持稳定文件名，不加日期前缀；
   - 非独一份文档（设计、调研、梳理、复盘、阶段性测试记录）使用 `YYYYmmdd-` 前缀；
   - 示例：`20260302-sync-continue-after-local-change.md`。
5. 设计目录化规则：
   - 若同一设计主题包含多个文件（如设计文档 + 原型 + 调试记录），必须创建主题目录统一收纳；
   - 目录建议命名为 `docs/design/YYYYmmdd-<topic>/`；
   - 同一主题的 `.md/.html/.txt/素材目录` 不应继续平铺在 `docs/design/` 根目录。

## 3. 归档与删除规则

1. 阶段性文档先放工作路径，满足以下任一条件即可归档到 `docs/archive/YYYY-MM/`：
   - 信息已并入权威文档；
   - 仅用于阶段性验证，后续不再持续更新；
   - 与主路径文档重复且无新增结论。
2. 默认不保留独立 bugfix/排障过程文档；如需临时记录，应在问题关闭后尽快合并进权威文档并删除临时文档。
3. 归档文档必须加“归档状态”说明，避免被误当作现行规范。
4. 删除策略：仅在归档后至少 1 个发布周期且无人引用时再评估删除。

## 4. 变更同步规则

1. CLI 参数变更：同步更新 `man/md-viewer.1` 与 `showHelp()` 输出。
2. 配置项变更：同步更新 `man/md-viewer.config.5` 与 `docs/配置说明.md`。
3. 用户可见行为变更：同步更新对应的设计文档/使用说明。
4. 测试策略或判定标准变更：同步更新 `docs/testing-spec.md`。

### 4.1 man 手册治理规则（借鉴 mt-cli）

1. `man/*.1` 与 `showHelp()` 保持“同一命令模型”：
   - 相同的子命令名与层级；
   - 相同的关键参数名（例如 `--limit/--offset/--json`）；
   - 相同的错误语义（参数错误 vs 运行时错误）。
2. `man/*.1` 采用稳定结构，避免自由发挥：
   - `NAME`、`SYNOPSIS`、`DESCRIPTION`、`GLOBAL OPTIONS`、`COMMANDS`、`EXAMPLES`、`FILES`、`SEE ALSO`
3. 行为真相源约束：
   - 用户侧命令行为以 `man/*.1` 为准；
   - 配置键与默认值以 `man/*.config.5` 为准；
   - `docs/配置说明.md` 只做说明与互链，不复制另一份独立参数真相。
4. 新增命令或子命令时，必须同时补齐：
   - `showHelp()` 示例；
   - `man/md-viewer.1` 对应命令段落与示例；
   - 若涉及配置，补 `man/md-viewer.config.5` 与 `docs/配置说明.md`。
5. 提交前最小人工校验：
   - `mdv --help` 与 `man/md-viewer.1` 的命令名/参数名一致；
   - 示例命令至少抽样执行 1 条，确认文案和真实行为一致。

## 5. 定期巡检节奏

1. 月度巡检（建议每月一次）：
   - 检查断链
   - 检查重复主题
   - 检查过时文档
   - 更新 `docs/README.md` 索引
2. 发布前巡检（每次发布建议执行）：
   - `man` 与配置文档同步性
   - 用户可见行为说明同步性
   - 归档目录是否有应上收为主文档的结论

## 6. 巡检执行清单（可复制）

```text
[ ] 运行文档链接检查（至少覆盖 docs/ 和 README.md）
[ ] 检查新增文档是否声明文档定位
[ ] 检查同主题是否出现多份“权威”文档
[ ] 检查可归档文档并迁移到 docs/archive/YYYY-MM/
[ ] 检查 man、配置说明与 showHelp() 是否一致
[ ] 检查 man 结构是否符合固定章节（NAME/SYNOPSIS/COMMANDS/EXAMPLES 等）
[ ] 更新 docs/README.md 索引
```

## 7. 文档提交前检查

1. 先执行：
   - `git status --short`
   - `git diff --name-only`
2. 确认改动范围仅包含目标文档，避免把无关文件混入同一提交。
3. 若涉及命令参数/配置行为，确认 `man/`、配置说明、`showHelp()` 已同步。
4. 提交信息建议使用 Conventional Commits，例如：
   - `docs(repo): add documentation governance spec`
   - `docs(sync): align sync workflow docs`
