# Tab 过多时的管理与关闭策略设计

文档定位（设计）：
- 负责定义 md-viewer 的 tab 管理策略、交互与状态规则。
- 不负责具体代码实现细节；实现以本设计为准增量落地。

## 背景

当前 `sessionFiles` 增多后，顶部 tab 条可读性快速下降：
- 标题挤压、重复文件名难区分。
- 关闭操作成本高，误关概率上升。
- 缺少“批量关闭”和“最近使用”能力。

目标：在不引入复杂页面的前提下，降低多文件协作时的视觉负担和误操作。

## 现状约束（基于当前代码）

1. 数据来源：`state.sessionFiles` + `state.currentFile`。
2. tab 渲染：`src/client/ui/sidebar.ts` `renderTabs()`，当前仅横向平铺。
3. 关闭动作：单个 tab 的 `window.removeFile(path)`。
4. 已有两种侧边栏模式：`simple/workspace`，可复用“列表侧管理”思路。

## 结论（先看）

1. 采用“动作在上、候选列表在下”的单层交互，不新增复杂页面。
2. 默认保持横向 tab 条，但增加“Tab 管理器”下拉面板作为集中管理入口。
3. 第一期先做低风险能力：搜索、按最近使用排序、批量关闭。
4. 分组与折叠采用“虚拟分组（最近使用/其他）”，不引入持久化树结构。

## 方案总览

### 1) 信息架构

- 顶部 tab 条：保留，用于快速切换。
- `Tab 管理器`（点击工具按钮展开，非 modal）：
  - 顶部操作区：
    - `关闭其他`
    - `关闭右侧`
    - `关闭未修改`
    - `关闭全部`
  - 中部过滤区：
    - 搜索输入（名称/路径）
    - 排序切换（最近使用/字母序）
  - 底部列表区：
    - 分组 A：最近使用（最近 N 个）
    - 分组 B：其他
    - 每项支持：切换、关闭、固定（后续）

### 2) 分组与折叠策略

- 分组是“视图分组”，不是数据结构分组。
- 默认折叠“其他”分组，减少初始噪音。
- 最近使用组默认展开，优先展示用户下一步最可能操作。

### 3) 批量关闭策略

动作定义：
1. 关闭其他：保留当前文件，关闭其余。
2. 关闭右侧：按 tab 可视顺序关闭当前右侧。
3. 关闭未修改：关闭 `M/D` 以外文件（保护风险项）。
4. 关闭全部：仅保留当前（避免空白状态抖动）。

保护规则：
- `D`（missing）文件优先展示，不默认被“未修改”规则关闭。
- `M` 文件属于风险态，不参与“关闭未修改”。

## 交互细节

### 入口

- 在现有 tab 条右侧增加 `管理` 按钮（例如 `[≡ Tabs]`）。
- 点击后展开面板；再次点击收起；点击外部关闭。

### 面板操作顺序

1. 用户先选择动作（上区）。
2. 中区即时展示受影响数量（如“将关闭 12 个”）。
3. 用户执行后，底部列表即时刷新。

### 失败与恢复

- 批量关闭失败（理论上很少）时：
  - 保持面板打开。
  - 在动作区下方显示错误文案。
  - 已成功关闭的不回滚（幂等优先）。
- 支持一次“撤销关闭”（后续可选，v2）。

## 数据与状态建议

新增轻量状态（前端内存 + localStorage）：
- `tabManager.open: boolean`
- `tabManager.query: string`
- `tabManager.sort: 'recent' | 'name'`
- `tabManager.collapseOthers: boolean`
- `tabAccessOrder: string[]`（MRU，切换文件时更新）

不变：
- 文件事实源仍是 `state.sessionFiles`。

## 分阶段落地

### Phase 1（建议本周）

1. 增加 `Tab 管理器` 面板。
2. 支持 4 个批量动作。
3. 支持搜索 + 最近使用排序。

### Phase 2

1. 固定 tab（Pin）。
2. 一键“仅保留固定 + 当前”。
3. 撤销关闭。

### Phase 3

1. 与 workspace 模式联动（按工作区过滤 tab 列表）。
2. 可选：会话快照（工作组）。

## 验收标准

1. 打开 30+ 文件时，用户可在 5 秒内定位并关闭目标集合。
2. “关闭未修改”不会误关 `M/D` 文件。
3. 面板关闭/打开与快捷切换不会阻断当前阅读流程。
4. 不新增 modal；交互全部在当前页面内完成。
