# 用户风格指南 (User Style Guide)

本文档总结用户在项目开发中的偏好和风格，帮助 AI agent 做出更符合用户期望的决策。

---

## 决策偏好

### 简单性 > 复杂性 (置信度: 95%)
- **描述：** 用户多次选择简单方案，拒绝复杂方案
- **示例：** 纯文本工具栏 > Linear 风格工具栏
- **理由：** 用户反馈"我甚至觉得你这个纯文本的视觉效果更好！"
- **应用：** 设计时优先考虑简单方案，避免过度设计

### 避免快捷键冲突 (置信度: 80%)
- **描述：** 用户倾向于避免添加快捷键
- **理由：** "省的和浏览器冲突"
- **应用：** 默认不添加快捷键，除非是独特的组合键
- **例外：** 如果是应用内独有的快捷键（不与浏览器冲突），可以考虑

### 先设计后实现 (置信度: 90%)
- **描述：** 用户要求设计类任务先出设计文档
- **格式：** Markdown 文档 + ASCII 原型
- **不要：** 直接出 HTML 原型（工具限制）
- **应用：** 涉及体验变化、复杂策略的任务，先写设计文档等待确认

### 实用主义 > 完美主义 (置信度: 85%)
- **描述：** 用户接受"足够好"的方案，不追求完美
- **示例：** 字体缩放功能无快捷键（虽然不完美，但够用）
- **应用：** 不要过度优化，功能可用即可

### 渐进式实施 (置信度: 90%)
- **描述：** 用户偏好分阶段实施（MVP → 优化 → 完善）
- **示例：** 多个设计文档都采用"阶段 1/2/3"的实施路线
- **应用：** 提供渐进式方案，先实现核心功能，后续按需优化

## 沟通风格

### 简洁直接 (置信度: 90%)
- **描述：** 用户喜欢简洁的回复，不喜欢冗长的解释
- **反例：** 过度解释技术细节
- **正例：** 直接给出方案和理由
- **应用：** 回复保持简洁，重点突出

### 视觉化 (置信度: 85%)
- **描述：** 用户喜欢 ASCII 原型和示例
- **示例：** 工具栏设计的 ASCII 原型
- **不要：** 只有文字描述
- **应用：** 设计方案中包含视觉示例

### 产品视角 > 技术视角 (置信度: 80%)
- **描述：** 用户更关注用户体验，而非技术实现
- **示例：** 关注"按钮是否突兀"，而非"CSS 实现细节"
- **应用：** 讨论方案时从用户体验角度出发

### 对比式决策 (置信度: 85%)
- **描述：** 用户喜欢看到多个方案对比
- **格式：** 方案 A / 方案 B / 方案 C + 优缺点 + 推荐
- **应用：** 设计文档中提供 2-4 个方案对比

## 质量标准

### 代码质量
- **风格：** 偏好简洁的代码结构（不过度抽象）
- **类型安全：** 使用 TypeScript 类型注解
- **函数式：** 偏好纯函数和不可变性
- **命名：** 清晰的命名（camelCase）

### 设计质量
- **一致性：** 统一的视觉风格
- **简洁性：** 不要花哨的动画和装饰
- **实用性：** 功能 > 美观
- **可访问性：** 考虑键盘导航、屏幕阅读器

### 文档质量
- **设计文档：** 设计类任务必须有文档
- **格式：** Markdown + ASCII 原型 + 代码示例
- **内容：** 方案对比 + 推荐 + 理由 + 实施路线
- **更新：** 完成后更新 TODO.md 和 tasks_dashboard.md

## 反模式（避免）

### 过度设计 (置信度: 95%)
- **描述：** 不要设计复杂的 UI（如 Linear 风格）
- **示例：** 用户拒绝了 Linear 风格工具栏，选择纯文本
- **应用：** 设计时优先考虑简单方案

### 生成原型工具 (置信度: 100%)
- **描述：** 不要尝试生成 HTML 原型（工具不支持）
- **替代：** 使用 ASCII 原型
- **示例：** 用户多次提醒"你需要换一个办法"

### 冗长的解释 (置信度: 80%)
- **描述：** 不要过度解释技术细节
- **应用：** 直接给出方案和理由，保持简洁

### 未经确认的重大变更 (置信度: 90%)
- **描述：** 不要在未经用户确认的情况下做重大变更
- **应用：** 设计类任务必须先出设计文档等待确认

### 破坏性操作 (置信度: 100%)
- **描述：** 不要使用 git reset --hard, force push 等破坏性命令
- **应用：** 优先使用安全的 Git 操作

### 重启开发服务器 (置信度: 90%)
- **描述：** 不要重启服务器（用户已经运行了 `bun run dev`）
- **应用：** 修改代码后，服务器会自动重载

## 任务处理风格

### 自主性 (置信度: 85%)
- **描述：** 用户希望 agent 自主完成任务
- **示例：** "你把剩下未开始的任务逐个都做一遍，遇到要做设计就做设计，设计完不要找我确认"
- **应用：**
  - 普通任务：完成后提交代码，继续下一个
  - 设计任务：完成设计文档，标记为等待，继续下一个
  - 不要频繁打断用户确认

### 批量处理 (置信度: 80%)
- **描述：** 用户希望 agent 批量处理多个任务
- **应用：** 一次会话中完成多个任务，而非一个任务一个会话

### 持续更新 (置信度: 90%)
- **描述：** 完成任务后立即更新 TODO.md 和 tasks_dashboard.md
- **应用：** 每完成一个任务，立即更新状态和统计

## 工具和技术栈偏好

### 开发工具
- **运行时：** Bun（不是 Node.js）
- **包管理：** Bun（不是 npm/yarn）
- **构建工具：** esbuild（客户端代码）
- **开发模式：** `bun run dev`（自动重载）

### 代码风格
- **语言：** TypeScript
- **格式化：** 使用项目配置（.editorconfig）
- **Lint：** 遵循项目规则

### Git 工作流
- **分支：** 直接在 main 分支工作（小项目）
- **提交：** 频繁提交，清晰的 commit message
- **格式：** `<type>: <description> (by Agent-ID)`
- **推送：** 完成任务后推送（可选）

## 项目特定约定

### 文件组织
- **设计文档：** docs/design/
- **任务文档：** TODO.md, docs/tasks_dashboard.md
- **客户端代码：** src/client/
- **服务端代码：** src/

### 命名约定
- **设计文档：** kebab-case.md（如 font-size-control.md）
- **任务 ID：** 无需显式 ID，使用任务标题
- **分支命名：** agent-{id}-task-{desc}（如需要 worktree）

### 任务状态
- **开始：** 默认状态（无标记）
- **进行中：** 🔄进行中 - Agent-ID - 日期时间
- **等待：** ⏳等待 - 日期
- **完成：** ✅完成 - 日期
- **放弃：** 🚫放弃 - 日期

## 使用说明（给 Agent）

### 决策时
1. 先查看本风格指南
2. 如果有明确偏好（置信度 > 80%），直接应用
3. 如果没有覆盖或置信度低，再询问用户

### 更新时
1. 每 10 次对话后，分析新的决策和反馈
2. 更新相关条目的置信度
3. 添加新发现的偏好
4. 删除过时或错误的条目

### 冲突时
1. 如果用户明确表达新偏好，立即更新风格指南
2. 如果与现有偏好冲突，询问用户确认
3. 记录例外情况（上下文相关的特殊决策）
