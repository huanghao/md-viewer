# 工作区排序交互设计（避免“拖入”误导）

## 背景问题
当前拖拽排序用“目标工作区整块高亮”，用户容易理解成“把 A 拖进 B（作为子项）”，而不是“在列表里重排顺序”。

目标：让排序意图清晰、低误解、可回退。

---

## 方案 A：插入线排序（推荐）

### 核心交互
- 拖拽开始：仅显示被拖项的半透明态。
- 拖拽经过其他项时：不高亮整块，只显示“插入线”。
- 插入线位置：
  - 经过目标项上半区：显示在目标项上方
  - 经过目标项下半区：显示在目标项下方
- 松手后按插入线位置重排。

### 为什么更准确
- “线”天然表达“插入到这里”，不会被理解为“放进这个容器”。
- 与 IDE、任务列表、看板的重排心智一致。

### 视觉稿（ASCII）

拖拽中：

```text
工作区

  case-cli
  md-viewer
  ─────────────────  <- 蓝色插入线（即将插入位置）
  notes
  blog
```

被拖项：
- 原位置保留占位（可选）
- 鼠标下跟随一个半透明副本（可选）

### 交互细节
1. 插入线左右对齐列表边界，不缩进到内容区。
2. 线宽 2px，颜色主色蓝，避免背景块误导。
3. 不允许“拖到自己”触发变化。
4. ESC 取消拖拽恢复原顺序（可选）。

---

## 方案 B：显式手柄 + 非拖拽区域不可拖

### 核心交互
- 每个工作区右侧显示 `⋮⋮` 手柄。
- 只有按住手柄才能拖动，点击标题仍是展开/切换。
- 拖动反馈仍使用方案 A 的“插入线”。

### 优点
- 显著降低误触（现在整行可拖很容易误触）。
- 用户一眼知道“这是排序功能”。

### 视觉稿（ASCII）

```text
case-cli      ⋮⋮
md-viewer     ⋮⋮
notes         ⋮⋮
blog          ⋮⋮
```

---

## 方案 C：无拖拽备选（键盘/按钮）

### 核心交互
- hover 显示 `↑` `↓` 按钮。
- 点击后上移/下移一位。

### 适用场景
- 触控板不习惯拖拽用户。
- 作为可访问性补充（键盘用户可聚焦后触发）。

---

## 推荐组合

推荐：`方案 A + 方案 B`，并预留 `方案 C` 作为补充。

1. 第一阶段（马上可做）
- 改成“插入线”反馈
- 移除整块高亮

2. 第二阶段（体验增强）
- 增加 `⋮⋮` 手柄，限制拖拽触发区域
- 增加 `↑/↓` 作为非拖拽替代

---

## 交互状态机（实现参考）

状态：`idle -> dragging -> drop-preview -> dropped/cancelled`

1. `idle`
- 无插入线
- 无拖拽样式

2. `dragging`
- 记录 `dragSourceId`
- 计算 `dropIndex`
- 渲染插入线

3. `dropped`
- 按 `dropIndex` 重排
- 持久化 `state.config.workspaces`
- 清理临时状态

4. `cancelled`
- 不改顺序
- 清理临时状态

---

## 验收标准

1. 用户不会把排序理解成“拖入工作区内部”。
2. 拖拽时始终可见明确插入位置。
3. 错误拖拽（拖到自身、拖出区域）不会改变顺序。
4. 排序结果刷新后保持。
