# PDF 结构化翻译 — 可行性分析

## 背景

当前翻译方案是逐段选择翻译，译文显示在右侧侧边栏。目标是探索"译文紧跟原文段落下方"的双语阅读体验，以及以此为基础的结构化文本层重建。

## 现有段落拆分机制

### 实现位置

`src/client/pdf-viewer.ts` — `buildTextBlocks`（约 712 行）

### 拆分逻辑

1. 过滤空 TextItem，按屏幕 Y 坐标排序
2. 相邻两项 Y 差 `< lineHeight * 1.5` → 同一段落，否则新段落
3. 坐标系：PDF 原生坐标（Y 从底部向上）→ 屏幕坐标（`screenY = viewportHeight / scale - pdfY`）

### 每个 block 包含的信息

```
{ pageNum, items[], text, x, y, width, height }
```

- `x/y`：段落左上角（屏幕坐标）
- `width/height`：边界框尺寸
- `items`：原始 PDF TextItem 数组（保留字号、坐标等原始信息）

### 已知弱点

- **双栏布局**：两栏文字 Y 坐标重叠，会被混合进同一段落。但双栏可以自动检测——X 坐标分布呈明显双峰，用页面中线切分即可分开处理，是可解决的问题。
- **脚注/页眉**：字号小但 Y 坐标接近正文，可能误合并。
- **标题紧接正文**：行间距不均匀时可能误切。
- **公式/表格等复杂排版**：数学符号高度异常，表格文字碎片化。需要先做真实数据分析，确认 badcase 比例，个人产品场景下如果占比低可以暂不处理。

## 三个方向的可行性分析

### 方向 A：Y 轴坐标切段落（改进现有拆分）

**可行性：高**

现有 `buildTextBlocks` 已经基于 Y 坐标拆分，效果不错。可以在此基础上增加：

- **双栏检测**：统计每页 TextItem 的 X 坐标分布，如果有明显双峰（两个 X 聚类中心），以中线为界分两列分别拆分。
- **字号突变检测**：字号 > 正文中位数 1.2x → 判定为标题，强制新段。
- **间距自适应阈值**：当前 1.5x 是全局固定值，可改为每页动态计算（基于行间距分布的众数）。

**工作量**：小，改 `buildTextBlocks` 即可，不影响其他功能。

---

### 方向 B：自建 Text Layer（完全重建文本结构）

**可行性：高，工作量最大**

放弃 PDF.js 原生 TextLayer，自己渲染文字层：

- Canvas 保留作为背景位图（不变）
- 文字层改为自定义 HTML，每段是一个 `<div>`，包含原文 + 译文子节点
- 段落之间真正 reflow，译文插入后后续段落自动下移
- 文本选中、评论锚点全部基于段落 index，不再依赖 PDF 坐标

**优点**：
- 双语阅读体验最佳（译文真正嵌入段落流）
- 文本选中、评论、高亮全部基于结构化段落，更稳定
- 可以做段落级别的折叠/展开译文

**缺点**：
- 需要重写渲染层，工作量大，需要完整设计后再动工
- 复杂排版（公式、表格、图注）需要 fallback 处理，先用真实 PDF 做数据分析确认规模

**关于字符级精确定位的问题**：

PDF.js 原生 TextLayer 会用 `transform` 矩阵把每个字符精确还原到 canvas 对应位置，实现像素级对齐。自建 text layer 放弃了这个能力，只做段落级对齐（把整段文字放在正确的 Y 位置）。

影响：文字不再与 canvas 背景的字形完全对齐，视觉上会有轻微错位。但对于翻译阅读场景，段落级对齐已经够用——用户看的是译文，不需要原文字符精确可选中。如果仍需要精确文本选中（如拖选原文做评论），可以保留原生 TextLayer 作为不可见的选中层，自建层只负责显示。

---

### 方向 C：Sidecar 存储结构

**可行性：高**

项目已有 `loadSidecar`（`src/client/toc-extractor.ts`），可以扩展为两个 sidecar 文件：

```
foo.pdf                      ← 原始 PDF
foo.pdf.structure.json       ← 段落结构（一次解析，永久缓存）
foo.pdf.translation.json     ← 翻译结果（按段落 index 存）
```

**结构 JSON 格式设计**：

```json
{
  "version": 1,
  "pages": [
    {
      "pageNum": 1,
      "blocks": [
        {
          "idx": 0,
          "text": "Abstract",
          "type": "heading",
          "x": 72, "y": 60, "width": 451, "height": 16
        },
        {
          "idx": 1,
          "text": "We present a novel approach...",
          "type": "body",
          "x": 72, "y": 82, "width": 451, "height": 48
        }
      ]
    }
  ]
}
```

**翻译 JSON 格式设计**：

```json
{
  "version": 1,
  "provider": "openai",
  "pages": {
    "1": {
      "0": { "text": "摘要", "timestamp": 1713456789 },
      "1": { "text": "我们提出了一种新颖的方法...", "timestamp": 1713456790 }
    }
  }
}
```

**优点**：
- 解析和翻译结果持久化，不需要每次重新解析
- 支持增量翻译（只翻译未翻译的段落）
- 结构 JSON 可以在服务端预生成，减少客户端计算

## 数据分析结论（2026-04-19）

工具：`scripts/pdf-para-lab/`，对 15 篇学术论文（805 页，13968 段）跑完整分析。

### 数字

| 指标 | 结果 |
|------|------|
| 总页数 | 805 |
| 总段落数 | 13968 |
| 平均段落/页 | ~17 |
| 双栏页面 | 375/805 (47%) |
| 可疑段落 | 1923 (13.8%) |

### 可疑段落构成

- **页码/页眉**（约 50%）：每页一个数字，PDF 本身结构，翻译时直接跳过
- **图表数字碎片**（约 35%）：坐标轴标签、图例数字，无法避免，翻译时跳过
- **表格行合并**（约 5-10%）：item > 60 的超大段落，是真正的拆分 badcase，每个 PDF 约 2-5 处

### 结论

段落拆分质量足够用。扣除页码和图表碎片后，真正影响翻译的 badcase 只有表格，占比极低。双栏检测逻辑有效（47% 页面正确识别为双栏）。

**不需要在拆分逻辑上继续投入**，可以直接推进方向 B（自建 text layer）的设计。

1. **翻译粒度**：按段落翻译 vs 按页翻译？段落翻译上下文不足，页翻译 token 消耗大但质量更好。
