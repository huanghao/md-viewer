# 评论重定位现状梳理（当前实现）

文档定位（调研）：
- 说明 `md-viewer` 当前版本里，文档内容更新后评论如何重定位、如何展示、会带来什么影响
- 本文描述“代码当前真实行为”，不等同于历史设计稿中的理想方案
- 设计目标另见 `docs/design/20260304-annotation-storage-and-anchoring.md`

## 背景

当前仓库里同时存在两类文档：
- 设计文档：描述想要的评论重锚能力，例如 fuzzy search、重锚日志、手动重绑
- 代码实现：已经上线的最小可用版本

如果不把两者分开，很容易误以为当前已经具备更复杂的重锚能力。本文只讨论当前代码已经实现的部分。

## 结论（先看）

1. 当前评论不是绑定在 DOM 节点、段落 ID 或行号上，而是绑定在一段文本上。
2. 重定位算法是“三段式”：原位置精确命中 -> 全文搜索原文 -> 多命中时用旧位置与前后文打分。
3. 当前没有实现模糊匹配、手动重绑、重锚日志，也没有版本级 diff 语义。
4. 文档更新后，评论大致会落到三类结果：`exact`、`reanchored`、`orphan`。
5. 已解决评论在重新计算锚点时会保留 `resolved` 状态，即使原文已经失锚，这里有理解成本，也有潜在 bug 风险。

## 1. 当前实现到底存了什么

创建评论时，前端会记录这些信息：

- `start`：评论创建时，在当前渲染纯文本中的起始偏移
- `length`：原选区长度
- `quote`：被评论的原始文本
- `quotePrefix`：原文前 32 个字符
- `quoteSuffix`：原文后 32 个字符
- `status`：`anchored` / `unanchored` / `resolved`
- `confidence`：当前重定位的置信度
- `thread`：评论主贴与回复线程

这意味着评论的“身份”不是“第 3 段第 2 行”，而是“这段原文及其附近上下文”。

- 选区采集：[src/client/annotation.ts](/Users/huanghao/workspace/md-viewer/src/client/annotation.ts#L1145)
- 存储模型：[src/annotation-storage.ts](/Users/huanghao/workspace/md-viewer/src/annotation-storage.ts#L14)

## 2. 重定位算法的实际计算方式

当前算法在 [src/client/utils/annotation-anchor.ts](/Users/huanghao/workspace/md-viewer/src/client/utils/annotation-anchor.ts#L39)。

### 2.1 第一步：原位置快速命中

系统先检查新的全文里，原来的 `start + length` 位置上，是否还是同一段 `quote`。

如果是：
- `status = anchored`
- `confidence = 1`
- UI 上会被视为 `exact`

这一步适合“评论附近没改，或者只改了评论之后的内容”的情况。

### 2.2 第二步：全文查找原文

如果原位置不再匹配，系统会在全文中查找 `quote`。

结果有两种：
- 只找到 1 处：直接重定位到那一处，`confidence = 0.8`
- 找到 0 处：标记为 `unanchored`

这一步适合“评论前面插了几段内容，导致偏移整体后移”的情况。

### 2.3 第三步：多命中时打分

如果同一个 `quote` 在全文里出现多次，系统不会随机选，而是给每个候选位置打分：

- 离旧 `start` 越近，分越高
- 前缀 `quotePrefix` 完全一致，加 500 分
- 后缀 `quoteSuffix` 完全一致，加 500 分

最后选分数最高的那个位置：
- `status = anchored`
- `confidence = 0.6`
- UI 上会被视为 `reanchored`

### 2.4 当前没有做什么

和设计稿相比，当前实现没有这些能力：

- 没有 fuzzy search
- 没有“旧位置附近窗口内模糊搜索”
- 没有手动重绑入口
- 没有重锚日志
- 没有文档版本或 diff 语义

这里补一层直白解释，避免这些词看起来像“只差一点点”的实现细节：

- `fuzzy search`
  指的是原文被轻微改写后，不再要求 `quote` 字面完全一致，而是允许按相似度去找“最像的句子”。
- “旧位置附近窗口内模糊搜索”
  指的是不在全文乱搜，而是先只在旧位置附近的一小段文本内做近邻搜索，更适合处理局部编辑。
- “手动重绑入口”
  指的是当系统失锚后，允许用户手动重新选中一段文本，把老评论重新挂到新位置。
- “重锚日志”
  指的是记录每次重定位前后发生了什么，例如“从原位命中降级为全文匹配”“从 anchored 变成 unanchored”。
- “文档版本或 diff 语义”
  指的是系统知道文档前后版本差异，例如某段文字是移动了、删除了还是替换了，而不只是拿当前全文做一次纯文本搜索。

所以它更像“基于原文与上下文的文本重定位”，不是“版本感知型评论系统”。

## 3. 一个实际好理解的例子

先看原始文档：

```md
# 发布流程

1. 构建产物
2. 上传 Homebrew tap
3. 更新 CHANGELOG
```

用户给 `上传 Homebrew tap` 加了一条评论。

系统实际保存的重点信息可以理解成：

```ts
{
  quote: "上传 Homebrew tap",
  start: 13,
  length: 16,
  quotePrefix: "1. 构建产物\n2. ",
  quoteSuffix: "\n3. 更新 CHANGELOG"
}
```

### 场景 A：前面插入了一段话

文档更新后：

```md
# 发布流程

发布前先确认本地测试通过。

1. 构建产物
2. 上传 Homebrew tap
3. 更新 CHANGELOG
```

影响：
- 原来的 `start=13` 已经失效
- 但全文里还能唯一找到 `上传 Homebrew tap`
- 评论会自动移动到新位置
- 最终效果：用户看到评论还挂在原句上，只是内部 `start` 被改写了

### 场景 B：文本重复出现

文档更新后：

```md
# 发布流程

预演：
2. 上传 Homebrew tap

正式：
1. 构建产物
2. 上传 Homebrew tap
3. 更新 CHANGELOG
```

影响：
- 全文里有两处相同的 `上传 Homebrew tap`
- 系统会更偏向旧位置附近、且前后文更像原文的那一处
- 如果上下文也变得很像，就有误绑风险

最终效果：
- 大多数情况下评论还能跟对
- 但重复句子越多、上下文越相似，误绑概率越高

### 场景 C：原文被彻底改写

文档更新后：

```md
# 发布流程

1. 构建产物
2. 推送软件源
3. 更新 CHANGELOG
```

影响：
- 全文里已经找不到 `上传 Homebrew tap`
- 评论内容与回复不会丢
- 但正文里不会再有对应高亮
- 该评论会进入 `unanchored`

最终效果：
- 评论还在
- 只是无法自动定位回正文

## 4. 这种方式在不同情况下的影响

### 4.1 小改动，效果通常较好

典型场景：
- 在评论前方插入一段内容
- 在评论后方插入一段内容
- 评论所在段落位置改变，但原句不变

影响：
- 评论往往能自动跟过去
- 使用体验接近“评论跟着内容走”

### 4.2 原文重复，存在误绑风险

典型场景：
- 列表项、模板句、规范用语大量重复
- 同一个段落被复制粘贴到别处

影响：
- 算法只能靠“旧位置距离 + prefix/suffix”来猜
- 如果候选项都很像，可能绑到错误位置
- 系统当前不会把这种“低确定性但已绑定”的结果单独标成风险态，只会显示 `reanchored`

### 4.3 原文被轻微改写，也会直接失锚

典型场景：
- `上传 Homebrew tap` 改成 `上传 tap`
- 给句子中间插入一个词
- 改了标点或空格，但改变了 `quote` 的字面值

影响：
- 因为当前算法依赖原始 `quote` 的精确搜索，不做模糊匹配
- 哪怕语义几乎没变，也可能直接从“可定位”变成 `unanchored`

这是当前实现相对保守的一面：
- 好处：不容易把评论错误地挂到看似相近但并非原文的位置
- 代价：对轻微改写不够鲁棒

### 4.4 富文本结构变化基本无关，文本流变化才重要

当前算法基于渲染后的纯文本流，不依赖 DOM 节点身份。

影响：
- 同样一句话从一个段落移到另一个段落，只要文本还能匹配，评论可能继续可用
- 纯样式变化、包一层标签、Markdown 渲染结构变化，只要最终文本流一致，通常不影响锚点
- 但如果渲染结果文本变了，还是会触发重定位或失锚

## 5. 不同评论状态的含义

当前实现里有两套相关概念。

### 5.1 持久化状态：`anchored` / `unanchored` / `resolved`

- `anchored`
  含义：当前能定位到正文
  用法：普通未解决评论的默认状态

- `unanchored`
  含义：当前找不到原文，评论实体仍保留
  用法：出现在“定位失败”筛选中，不会在正文里有对应高亮

- `resolved`
  含义：线程被用户标记为已解决
  用法：和“是否能定位”不是一个维度，但当前实现把它们压在同一个字段里

代码对应：
- 状态合并：[src/client/annotation.ts](/Users/huanghao/workspace/md-viewer/src/client/annotation.ts#L498)

### 5.2 UI 轨迹状态：`exact` / `reanchored` / `orphan`

这是界面根据 `status + confidence` 推导出来的显示态，不是单独存库字段。

- `exact`
  条件：`status != unanchored` 且 `confidence >= 0.95`
  含义：还在原位置精确命中

- `reanchored`
  条件：`status != unanchored` 且 `confidence < 0.95`
  含义：系统自动换了位置，或者至少不是原位精确命中

- `orphan`
  条件：底层状态是 `unanchored`
  含义：这是 UI 对“失锚评论”的展示名

这里故意保留两个词，但它们不是同一层概念：

- `unanchored` 是持久化状态，存到数据里的词
- `orphan` 是界面展示词，告诉用户“这条评论当前找不到正文位置”

如果只看产品语义，它们表达的是同一件事；如果从代码分层看，一个是数据状态，一个是 UI 轨迹名。

代码对应：
- UI 推导：[src/client/annotation.ts](/Users/huanghao/workspace/md-viewer/src/client/annotation.ts#L328)

## 6. 目前这些状态是怎么被使用的

### 6.1 筛选逻辑

默认筛选是 `open`，而 `open` 会排除掉 `resolved` 和 `orphan`。

这带来的直接效果是：
- 一条评论失锚后，默认列表里可能直接“消失”
- 它其实还在，只是被归到了 `orphan`

代码对应：
- 筛选逻辑：[src/client/annotation.ts](/Users/huanghao/workspace/md-viewer/src/client/annotation.ts#L334)

### 6.2 正文高亮逻辑

每次文档刷新、切换或重新渲染后，系统都会重新计算所有评论位置，并把成功定位的评论重新画到正文里。

代码对应：
- 重算与回写：[src/client/annotation.ts](/Users/huanghao/workspace/md-viewer/src/client/annotation.ts#L951)
- 文件刷新触发：[src/client/main.ts](/Users/huanghao/workspace/md-viewer/src/client/main.ts#L170)

### 6.3 已解决评论的特殊行为

当前状态合并规则是：

- 如果当前已经是 `resolved`
- 那么无论本次重算结果是 `anchored` 还是 `unanchored`
- 都继续保持 `resolved`

测试也明确验证了这个行为：
- [tests/unit/annotation-status.test.ts](/Users/huanghao/workspace/md-viewer/tests/unit/annotation-status.test.ts#L5)

这说明当前系统实际上把“线程是否已解决”放在了比“锚点是否还有效”更高的优先级上。

## 7. 可能的 bug 风险与认知风险

## 7.1 `resolved` 会遮蔽失锚事实

这是当前实现里最值得关注的一点。

现状：
- 已解决评论即使原文已经不存在，也不会转成 `unanchored`
- UI 的 `orphan` 判断依赖 `status === 'unanchored'`

影响：
- 已解决但实际已经失锚的评论，不会进入 `orphan` 视图
- 用户可能误以为“已解决评论仍然定位正常”
- 系统也失去了“已解决但已过期”的可见性

这更像是一个建模问题：
- `resolved` 是线程工作流状态
- `anchored/unanchored` 是锚点健康状态

二者本来应该拆开存，而不是复用一个字段。

评论里提到一个可能的反方向解释：既然已解决评论通常不再频繁关注，是否可以不再重锚，以减少计算量。

这个想法在性能上不是完全没道理，但它解决的是“要不要继续算”的问题，不是“字段是否应该复用”的问题。两者需要分开看：

- 如果为了性能，选择“已解决评论默认不参与重锚”，这是计算策略
- 如果把 `resolved` 和 `unanchored` 塞进同一个字段，这是数据建模

即使未来决定对已解决评论少算或不算，线程状态和锚点状态仍然最好拆开。否则一旦后续又需要：

- 展示“已解决但已失锚”
- 统计“失锚率”
- 支持 reopen 后恢复锚点健康检查

就会继续遇到语义互相覆盖的问题。

## 7.2 轻微改写会产生假阴性

现状：
- 只做精确文本匹配，不做模糊匹配

影响：
- 原意没变的小修订，可能直接被判成失锚
- 用户会觉得系统“太脆弱”

这不是传统意义上的 bug，但会造成体验偏差。

## 7.3 重复文本会产生误绑

现状：
- 多命中时只靠距离和 prefix/suffix 打分

影响：
- 相似段落、模板化文档里，可能自动挂错地方
- 这类错绑比“明确失锚”更危险，因为它看起来是正常的

这也是当前实现的主要质量风险之一。

## 7.4 置信度语义较粗

当前 `confidence` 只有三档典型值：
- `1`
- `0.8`
- `0.6`

影响：
- 它更像轨迹分类标签，不像真正连续概率
- 后续如果想根据置信度做排序、提醒或阈值控制，表达力有限

## 7.5 计划能力与现状不一致的认知风险

设计文档曾提到：
- 近邻模糊搜索
- 手动重绑
- 重锚日志

但这些能力当前并未实现。

影响：
- 读设计稿的人容易高估当前能力
- 讨论问题时，如果不区分“已实现”和“计划中”，容易误判 bug 或误判预期

## 8. 当前实现的总体评价

从工程取舍看，当前方案是一个合理的 v1：

- 比纯 `start/length` 偏移稳定很多
- 不需要引入版本图和 diff 模型
- 小改动场景下效果通常足够好

但它也明确有边界：

- 不擅长处理轻微改写
- 对重复文本存在误绑风险
- 将 `resolved` 与锚点状态共用一个字段，语义不够干净

如果后续继续迭代，优先级更高的改进方向应是：

1. 把“线程状态”和“锚点状态”拆开
2. 为失锚和低置信度重绑提供更明确的 UI 提示
3. 引入受控的近邻模糊匹配，而不是只做全文精确搜索
